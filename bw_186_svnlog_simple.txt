issue ID: CSOL-7496 
程序内容需求——配置好的光效碰撞不能达到预期效果。 
添加新的碰撞算法。
调用BigWorld.isCollide(bool u1,bool u2,int ei,int e2)。
其中，u1和u2对应两个entity使用的包围盒，若为False则表示使用entity本身的boundingbox，即一个AABB包围盒。若为Truw则表示使用一个OBB包围盒。
e1是一个entity的id，e2是另一个id。
当u1或u2为True时，当entity包含多个模型时，则只使用第一个模型的包围盒做为包围盒。
返回为True则碰撞了。若返回false，则表示有可能找不到ID对应的entity，或者该entity并没有可供使用的碰撞属性。

issue ID: CSOL-6449 BUG - 粒子编辑器中粒子叠加问题
为meta_particle_system（MPS）新增了一个选项，具体可以在粒子编辑器中点选对应的xml可打开对应的控制面板，可以控制该选项。
该选项将决定MPS内的particle_system(PS)的渲染顺序。当该选项为FALSE时，按照原先的方式对距离进行排序后渲染，当选项为TRUE是，整个MPS将会生成一个新的particle(P)，该P将包含原有的部分PS，其距离为所有PS的距离平均值。P将按xml内部的配置顺序渲染PS。
该修改将会修改MPS的xml文件的部分格式。用旧编辑器版本打开该新编辑器保存的xml文件时，将会导致错误。

issue ID: CSOL-5175 Bug——小地图显示异常

异常的原因：
小地图的多线程加载机制不完善，所以造成了莫名的黑屏和透明。
现优化和完善了小地图的多线程加载机制。

原本的加载流程：
在脚本上将6*6个CHUNK的DDS的文件地址直接传到底层，在底层中直接传给后线程，在后线程中全部加载后，再混合到一个数组中，最后在传给主线程，上传到显存。
现在的加载流程：
在脚本上将7*7个chunk的dds的文件地址传到底层，其中6*6个是主要chunk，剩下的是后备chunk。主要chunk直接在主线程中处理，先是查看是否保存有内存数据，若没有则直接读取，然后查看是否有显存数据，若没有则直接上传，最后再将所有的显存数据在GPU中合成一张大的纹理，然后再输出。上传的显存数据会被保留，等下一次的调用。后备chunk则先查看有没有内存数据，若没有，则放到后线程中进行加载，加载完毕后，则直接保存到主线程中等待调用。

由于小地图的特殊性，只有每次更换大地图的时候需要在主线程中进行大量文件IO外，其余时间文件IO均在后线程中完成。

新增SimpleCamera，创建方法BigWorld.SimpleCamera()

脚本方法：
shake(float duration, Vector3 amount, Vector3 posAmout = Vector3(0.f, 0.f, 0.f)) 震屏效果

脚本读写属性：
roll：roll旋转值， float，默认值（0）
dirInWS：摄像机镜头的朝向， Vector3， 默认值（ 0， 0， 1）
posInWS：摄像机的位置， Vector3， 默认值（0， 0， 0）
uprightDir 摄像机上方的朝向， Vector3，默认值 （0， 1， 0）
updateCallback 绑定update回调函数， 函数格式 def update(dTime)

例如：控制摄像机移动的用法

simpleCram = None
speed = Vector3(0, 0, 1)

def updata(dTime):
   global simpleCram, speed
   simpleCram.posInWS += speed * dTime

simpleCram = BigWorld.SimpleCamera()
simpleCram.updateCallback = updata
BigWorld.camera(simpleCram)

issue_id: CSOL-6126 微软拼音2007输入法，使用“- =”寻找汉字时，符号会直接录入输入框。
原因：微软拼音2007输入法在选取汉字时，没有完全捕获“- =”的WM_KEYDOWN消息， 而客户端进程的InputHandle模块处理WM_KEYDOWN消息时没有判断是否处于IME状态，重复处理“- =”。其他常用的中文输入法和微软拼音2003不会出现此情况。

解决方案：处于IME状态时，客户端进程的InputHandle模块不捕获WM_KEYDOWN消息。

Revision: 308
Author: support
Date: 2009年4月3日 13:51:38
Message:
Some GUI objects using alpha blending have been added into BW and they are added to the SortedChannel. The SortedChannel is used to draw transparent objects. Problem arises when those GUI objects are used with shimmer. Usually GUI is drawn after shimmer so shimmer will not displace GUI. But as those objects are drawn with other alpha objects, they are displaced.
To fix this, I have added a new channel named UnfilteredSortedChannel and put all transparent GUI stuffs into it. It will be drawn after shimmer and before all other GUI component.
The potential problem is, as alpha objects must be drawn sorted and those GUI objects has there depth in world. So if they are covered by another transparent object that is close to the camera, the pixles in the common part should be blended. However, in this case they will not be blended correctly as we draw them in the wrong order.

Message:
issue id：CSOL-2262

新增 SimpleGUIComponent 色彩混合功能

1)加入颜色混合的属性 
   调节亮度因子：colourLightFactor， 类型 Vector4， 默认值： （0.8， 0.8， 0.8， 1.0） 
                它的元素：[x, y, z, w], 元素类型：float，元素的取值范围 0.0～1.0 
                xyzw 分别对应于像素色彩的 RGBA；其中w值为0：全透明， 为1：非透明 
   调节RGB混合比例因子： colourBlendFactor， 类型 Vector3， 默认值：(0.3,0.59,0.11) 
                它的元素: [x, y, z]， 元素类型： float：元素的取值范围 0.0～1.0 
                xyz 分别对应与像素的色彩的RGB 

计算公式： 输入像素点colourInput， 输出像素点colourOutput 
float tmp = colourInput.r*colourBlendFactor.x + colourInput.g*colourBlendFactor.y + colourInput.b*colourBlendFactor.z 
          colourOutput = (tmp*colourLightFactor.x , tmp*colourLightFactor.y ,tmp*colourLightFactor.z, colourInput.a*colourLightFactor.w) 

(2) 新增materialFX属性: COLOUR_EFF 
    通过设置SimpleGUIComponent 类及其派生类的materialFX为 “COLOUR_EFF”， 可以启用界面颜色混合效果。 默认效果：非透明，0.8亮度的灰色效果

修复 CSOL-2399  碰撞检测失败的Bug。
http://172.16.0.234/browse/CSOL-2399
bug出现的原因：之前的碰撞检测算法过于简单（但很高效），碰撞用到三角及其检测点刚好无法检测 角色右边 的向底部倾斜度三角形（特例） 
解决方案： 由原来角色用到的一个碰撞三角形改为两个碰撞三角形（组成一个四边形），并在碰撞三角形的每个顶点和中心点作为采用点，检测前面是否有障碍物。修改算法后，可以避免模型几何不规则而造成的碰撞检测失败。

Author: yangjing
Date: 2008年10月23日 16:07:52
Message:
bigbang 工程无法打开datas\universes\zly_bi_shi_jian_bad 空间
问题原因查找和解决办法

原因：

由于程序初始化的时候，相机的中心点在（0，2，0）位置，如果文件中有space.localsetting文件,由于程序中产生一个ChunkDirMapping 对象
而此对象有一个XZ平面上的一个范围，如果相机不在这个范围内。就无法生成 CameraChunk 对象 ，程序始终在相机附近寻找在此范围内的Chunk对象直到找到为止。显然不可能找到

方法：


首先判断相机中心位置，是否在ChunkDirMapping 对象
规定的一个XZ平面上的一个范围内，如果不在这个范围内，让相机的中心位于此范围的中心
